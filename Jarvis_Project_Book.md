## 1. מבוא
### 1.1 רקע כללי
בעידן הדיגיטלי המודרני, משתמשי מחשב עובדים שעות רבות מול הדפדפן לצורכי קריאה, לימוד, כתיבה ועבודה מקצועית עם טקסטים ומסמכים. במקרים רבים, על מנת לקבל עזרה מכלי בינה מלאכותית (LLMs), נדרש המשתמש לבצע פעולות העתק-הדבק ידניות וחוזרות ונשנות, לעבור בין טאבים שונים ולנהל בצורה מפוזרת את המידע שנוצר. תהליך זה פוגע ברצף העבודה וביעילות שלה.
מערכת J.A.R.V.I.S Web Assistant פותחה על מנת לגשר על פער זה. המערכת מורכבת מתוסף דפדפן חכם (Chrome Extension) הפועל ברקע ואתר ניהול מרכזי (React UI). המערכת מאפשרת למשתמש לסמן טקסט בכל אתר אינטרנט, להפעיל עליו מניפולציות בינה מלאכותית (סיכום, שכתוב, הסבר, ניתוח דקדוקי) ולנהל את כל היסטוריית העבודה תחת ממשק אחד מרוכז, מאובטח ונוח.

## 2. מטרות המערכת
### 2.1 מטרת הפרויקט
לתכנן, לפתח ולהטמיע מערכת עוזר דיגיטלי אינטגרטיבית המבוססת על מודלי שפה וראייה ממוחשבת מתקדמים (Google Gemini ו-Groq Whisper). המערכת מאפשרת אינטראקציה חלקה עם טקסטים, תמונות והקלטות קוליות ישירות מהדפדפן, ללא צורך בהגדרות צד-לקוח מורכבות או חשיפת מפתחות API רגישים.

### 2.2 היבטים לשיפור בתהליך העבודה
המערכת משפרת את השימוש בכלי בינה מלאכותית במספר היבטים מרכזיים:
1. **ייעול תהליכים:** ביטול הצורך בהעתקה ידנית של טקסטים ומעבר לאתרים חיצוניים.
2. **גישה מהירה:** תפריט הקשר של תוסף הדפדפן מאפשר הפעלה מיידית של משימות מוגדרות מראש (סיכום, הסבר ושכתוב).
3. **תיעוד וניתוח:** ריכוז אוטומטי של כל היסטוריית השיחות במסד נתונים מאובטח, עם אפשרות למחיקה וניתוח סטטיסטי.
4. **עבודה מולטי-מודאלית:** הרחבת יכולות העוזר הדיגיטלי לתמיכה בניתוח ממשקי משתמש (תמונות מסך) ותמלול הקלטות קוליות.

### 2.3 סקירת מצב השוק
כיום קיימים תוספי דפדפן שונים ואתרי צ'אט כלליים לשימוש ב-AI. עם זאת, מרבית הכלים דורשים תשלום קבוע, שומרים את המפתחות של המשתמש בענן ללא בקרה, או שאינם משלבים בין היסטוריית פעולות מלאה לבין תוסף דפדפן קל משקל שאינו מכביד על חומרת המחשב. פרויקט זה מספק פתרון ייחודי על ידי שימוש בארכיטקטורת **BYOK (Bring Your Own Key)** דרך שרת Proxy מאובטח, ובכך פותר את בעיית עלויות הרישוי והאבטחה של צד שלישי.

## 3. תיאור התוכנה
### 3.1 תכנון המערכת ורכיביה
המערכת בנויה משלושה חלקים עיקריים המשלימים זה את זה:
1. **תוסף הדפדפן (Chrome Extension):**
   * תומך ב-Manifest V3 ומאפשר זיהוי וסימון טקסטים בכל דף אינטרנט.
   * שולח את המידע לעיבוד בשרת צד שלישי ומציג את התוצאות בחלון קופץ (Popup HUD) מהיר.
   * מסתנכרן אוטומטית עם סטטוס החיבור של המשתמש באתר הניהול.
2. **אתר הניהול (React Dashboard):**
   * מערכת כניסה והרשמה מאובטחת המבוססת על JWT.
   * ממשק הגדרות אישיות (Settings) לבחירת שפה מועדפת והזנת מפתחות API אישיים של Gemini/Groq.
   * לוח בקרה סטטיסטי המציג את כמות השיחות, זמני שימוש ופילוח פעולות.
   * מסופים ייעודיים לניתוח מסמכים (Document Review), תמלול קבצי קול (Audio Terminals), וניתוח ממשקי משתמש (Screen Analysis).
3. **שרת Backend (Node.js & Express):**
   * אחראי על ניהול המשתמשים ואימותם (Bcrypt & JWT Middleware).
   * משמש כצינור תקשורת מאובטח (Proxy) המעביר את הבקשות ל-APIs של Google Gemini ו-Groq, מבלי לחשוף את מפתחות ה-API בדפדפן.
   * מתעד את היסטוריית הפעולות ושומר אותה במסד הנתונים MongoDB Atlas.

## 5. ארכיטקטורה ופונקציונליות
### 5.1 ארכיטקטורת המערכת
המערכת מבוססת על ארכיטקטורת **Client-Server** מנוהלת ומחולקת לשלוש שכבות לוגיות:
1. **שכבת התצוגה (Presentation Layer):** נכתבה באמצעות **React (Vite)** ויוצרת ממשק משתמש דינמי מבוסס רכיבים (Components). אליה מצטרף תוסף הדפדפן הפועל ישירות בתוך הטאבים של המשתמש.
2. **שכבת הלוגיקה (Application Logic Layer):** שרת **Node.js/Express** עצמאי. השרת מקבל את הבקשות, מריץ בדיקות אבטחה, ומנהל את הלוגיקה מול ה-APIs של Google Gemini ו-Groq.
3. **שכבת הנתונים (Data Layer):** בסיס נתונים בענן **MongoDB Atlas**. השרת מתקשר עם בסיס הנתונים באמצעות **Mongoose ODM** כדי לשמור את נתוני המשתמשים והיסטוריית הפעולות בצורה מאובטחת.

### 5.2 טופולוגיית המערכת (העברת הבקשות)
מבנה מעבר הנתונים במערכת מתבצע באופן הבא:
```text
[משתמש מסמן טקסט בדפדפן] 
          │
          ▼
[תוסף הדפדפן (Chrome Extension)] ◄─── (קורא את ה-JWT והמפתחות מסנכרון הלקוח)
          │
          ▼ HTTP POST (/api/ai/process)
[שרת ה-API (Express Server)] ◄─── (מאמת את ה-JWT ב-Middleware)
          │
          ├─────────────────────────┐
          ▼ (חיבור זמני ב-RAM)       ▼ (תיעוד השיחה)
[APIs של Gemini & Groq]       [מסד נתונים MongoDB Atlas]
```

### 5.3 בסיס הנתונים וסכמות (MongoDB Atlas)
בסיס הנתונים מורכב משתי קולקציות מרכזיות המקושרות ביניהן:

#### סכמת משתמש (`User` Schema):
```javascript
{
  email: { type: String, required: true, unique: true },
  password: { type: String, required: true },
  createdAt: { type: Date, default: Date.now }
}
```

#### סכמת שיחות והיסטוריה (`Conversation` Schema):
```javascript
{
  userId: { type: String, required: true }, // מזהה המשתמש שביצע את הפעולה
  actionType: { type: String, required: true }, // סוג הפעולה: summarize, explain, rewrite, vision, transcribe
  selectedText: { type: String, required: true }, // הטקסט המקורי שנשלח לעיבוד
  aiResponse: { type: String, required: true }, // התשובה שהתקבלה מהמודל
  createdAt: { type: Date, default: Date.now }
}
```

### 5.4 טכנולוגיות בשימוש
* **Frontend:** React, HTML5, CSS3, Vite, Recharts, Lucide Icons.
* **Extension:** Manifest V3 JavaScript.
* **Backend:** Node.js, Express.js.
* **Database:** MongoDB Atlas (Cloud).
* **Authentication:** JSON Web Tokens (JWT).
* **AI Client Library:** `@google/generative-ai`, `groq-sdk`.

### 5.5 שפות הפיתוח
* **JavaScript (ES6+):** משמשת כשפת הפיתוח המרכזית הן בצד הלקוח (React & Extension) והן בצד השרת (Node.js).
* **HTML5 / CSS3:** ליצירת המבנה והעיצוב העתידני של המערכת.

### 5.6 סביבת השרת
השרת מוגדר כשרת API מבוסס פרוטוקול HTTP/HTTPS, המאזין כברירת מחדל לפורט `5000`. השרת מנהל את הניתובים באמצעות Express Router ומטפל בנתוני קלט בפורמט JSON (כולל הגבלת נפח של עד 50MB לצורך תמיכה בהעלאת קבצי תמונות מסך ואודיו ב-Base64).

### 5.7 ארגון ומבנה תיקיות הפרויקט
מבנה הפרויקט מחולק לשלוש תיקיות עצמאיות:
* `/client` - מכיל את קוד המקור של אתר הניהול ב-React (דפים, רכיבים, עיצובים, Context).
* `/server` - מכיל את שרת ה-Express, מודלי הנתונים (Models), הניתובים (Routes), וספריית ה-Middleware.
* `/extension` - מכיל את קובץ ה-Manifest, קובצי ה-JS של ה-Background Worker וה-Content Scripts ותפריט ה-Popup.

### 5.8 אמצעי אחסון
* **אחסון נתונים (MongoDB Atlas):** שמירה קבועה של פרטי משתמשים והיסטוריית פעולות.
* **אחסון זמני (Temp System Storage):** בעת ביצוע תמלול אודיו, קובץ ה-Base64 מומר זמנית לקובץ שמע מקומי בתיקיית הטמפ' של השרת, נשלח ל-Groq Whisper, ונמחק מיידית באמצעות פקודות מערכת הקבצים (`fs.unlinkSync`).

### 5.9 סביבת הטמעה ושימוש
1. המשתמש פותח את האתר, נרשם ונכנס לחשבון.
2. ה-Content Script סורק את האתר, מזהה את ה-JWT, ומעביר אותו לתוסף הדפדפן.
3. המשתמש מגדיר את מפתחות ה-API שלו בדף ההגדרות באתר. המפתחות נשמרים בדפדפן.
4. המשתמש מסמן טקסט בכל אתר, פותח את חלון התוסף או לוחץ קליק ימני ובוחר בפעולה רצויה.
5. התוצאה מוצגת בתוסף ונשמרת אוטומטית בהיסטוריית ה-Dashboard.

### 5.10 טופולוגיית ההטמעה והשימוש
ממשקי התוסף והאתר פועלים במקביל מול שרת ה-API. במידה והמשתמש מבצע מחיקת היסטוריה מהאתר, השינויים משתקפים מיד במסד הנתונים ומתעדכנים בכלל הממשקים בזמן אמת.

## 6. אבטחת מידע
### 6.1 יעדים עיקריים
* מניעת חשיפת מפתחות ה-API של המשתמשים לשרת ה-API שלנו או לגורמים חיצוניים (BYOK).
* אבטחת היסטוריית השיחות של כל משתמש כך שרק הוא יוכל לגשת אליה.
* הצפנת סיסמאות משתמשים למניעת פריצות למסד הנתונים.

### 6.2 נכסים מרכזיים
* סיסמאות משתמשים (נשמרות מוצפנות באמצעות Bcrypt).
* מפתחות API אישיים (נשמרים מקומית בדפדפן ומועברים ב-Header מוצפן ב-HTTPS בלבד).
* היסטוריית פעולות המשתמשים.

### 6.3 גורמי סיכון ואיומים
* **גניבת מפתחות API:** במידה ומפתחות ה-API היו נשמרים במסד הנתונים, פריצה למסד הייתה חושפת את מפתחות המשתמשים.
* **התקפות Brute Force:** ניסיונות התחברות חוזרים ונשנים לחשבונות משתמשים.
* **שינוי נתונים על ידי משתמש אחר (ID Spoofing):** ניסיון למשוך היסטוריה של משתמש אחר באמצעות שינוי ה-ID בבקשה.

### 6.4 אסטרטגיית ההגנה
1. **ארכיטקטורת BYOK Proxy:** השרת מעולם לא שומר את מפתחות ה-API בדיסק או במסד הנתונים. הוא משתמש בהם בזיכרון ה-RAM הזמני בלבד לצורך ביצוע הקריאות.
2. **אימות מבוסס JWT:** כל ניתוב רגיש בשרת מוגן באמצעות Middleware המאמת את ה-Token של המשתמש.
3. **הפרדת משאבים מוחלטת:** בעת שליפת היסטוריית השיחות, השרת משתמש במזהה המשתמש (`userId`) שחולץ ישירות מתוך ה-JWT המוצפן, ולא מתוך גוף הבקשה, מה שמונע זיופים.

### 6.5 אמצעי אבטחה בפועל
* שימוש ב-Bcrypt ליצירת Salt של 10 מחזורים והצפנת סיסמאות.
* הגבלת תוקף ה-JWT לצורך מניעת שימוש ב-Tokens ישנים במקרה של זליגה.
* תעבורת נתונים מאובטחת באמצעות פרוטוקול HTTPS.

### 6.6 בעיות אפשריות בתהליך ההזדהות
* זליגת ה-JWT מחשב המשתמש (למשל במחשב ציבורי). המערכת כוללת כפתור Logout מובנה המוחק לחלוטין את ה-Token מ-localStorage של האתר ומ-chrome.storage של התוסף.

### 6.7 טופולוגיית פתרון האבטחה
מנגנון ה-Proxy מונע חדירת קוד עוין לשרת ומספק הגנה מירבית על מפתחות המשתמש.

## 7. בדיקות המערכת
### 7.1 בדיקות תהליכיות ברמת המשתמש (End-to-End)
כדי לוודא שהמערכת פועלת בצורה תקינה, הורצו הבדיקות הבאות באופן ידני:
1. **בדיקת הרשמה וכניסה:** יצירת משתמש חדש, הזנת סיסמה לא נכונה (אימות כי המערכת חוסמת), כניסה מוצלחת ומעבר ל-Dashboard.
2. **בדיקת סנכרון התוסף:** כניסה לאתר, פתיחת התוסף ואימות כי הודעת ה-`Please Login First` נעלמת ובמקומה מוצג חלון העבודה.
3. **בדיקת קריאת AI (טקסט):** סימון טקסט בדף אינטרנט, לחיצה על "סכם" בתוסף, וקבלת תשובה מדויקת בתוך פחות מ-2 שניות.
4. **בדיקת תמלול (אודיו):** הקלטת קטע קצר במסך recordings, לחיצה על כפתור תמלול, וקבלת התמלול ומילות המפתח כתוצאת JSON מהשרת.
5. **בדיקת ניתוח ויזואלי (Vision):** העלאת תמונת מסך של ממשק משתמש בדף Screen Analysis, שליחה לעיבוד, וקבלת פירוט שגיאות ודירוג ניגודיות.
6. **בדיקת מחיקה מהיסטוריה:** לחיצה על כפתור מחיקה של שורה בטבלת השיחות, ואימות כי השורה נמחקה ואינה מופיעה לאחר ריענון.

### 7.2 בדיקות יחידה למודולים המרכזיים
* **בדיקת שרת ה-API:** שימוש ב-Postman לצורך בדיקה שכל פנייה ישירה לשרת ללא Header של JWT נחסמת עם קוד שגיאה `401 Unauthorized`.
* **בדיקת ה-AI Middleware:** שליחת בקשה לשרת ללא מפתח Gemini/Groq ואימות קבלת שגיאה מתאימה המנחה את המשתמש להגדיר מפתח.

## 9. ממשק לקוח ומסכי המערכת
להלן פירוט מקיף של כלל מסכי האפליקציה שהוטמעו בפועל במערכת J.A.R.V.I.S:

### 9.1 עמוד בית / התחברות והרשמה (Login / Register)
ממשק עתידני המציג את תיאור המערכת וטופס כניסה/הרשמה בעל עיצוב Stark Industries. המשתמש מזין אימייל וסיסמה. בכניסה מוצלחת, המערכת מקבלת JWT ושומרת אותו ב-localStorage.

### 9.2 לוח בקרה ראשי (Dashboard Terminal)
המסך הראשי של המערכת המציג נתונים סטטיסטיים חיים על השימוש של המשתמש:
* כרטיסיות נתונים המציגות: כמות שיחות כוללת, זמן עבודה שנחסך (במילים/דקות), כמות מילים מעובדות.
* גרף עמודות/קווים אינטראקטיבי (Recharts) המציג את כמות קריאות ה-AI המבוצעות בכל יום בשבוע האחרון.
* רשימת פעילויות אחרונות וחיבורים למערכת.

### 9.3 עמוד היסטוריית פעולות (History Logs HUD)
מסוף המציג טבלה מלאה של כל קריאות ה-AI שבוצעו על ידי המשתמש (הן מהאתר והן מהתוסף):
* הטבלה מציגה את סוג הפעולה (סיכום, שכתוב, ויז'ן, תמלול), הטקסט המקורי, והתשובה שהתקבלה.
* אפשרות חיפוש מהירה לפי מילים בתוך הטקסט.
* כפתור מחיקה מהיר לכל שורה המאפשר ניקוי רשומות רגישות ממסד הנתונים.

### 9.4 תוסף הדפדפן (Chrome Extension Popup)
חלון העוזר הדיגיטלי המינימליסטי המופעל ישירות מהדפדפן:
* מציג את הטקסט שסומן על ידי המשתמש בדף האינטרנט.
* כפתורים מוגדרים מראש: `Summarize` (סכם), `Explain` (הסבר), `Rewrite` (שכתב).
* אזור תצוגת תשובת ה-AI מבוסס אנימציית טקסט המדמה את פעילות המחשב המרכזי.

### 9.5 עמוד הגדרות מערכת וקינפוג מפתחות (Settings)
דף התצורה המרכזי של המערכת:
* בחירת שפת ממשק מועדפת (עברית, אנגלית, ספרדית, צרפתית).
* בחירת ערכת נושא UI (עתידנית, בהירה, שחורה).
* שדות קלט מוצפנים להזנת מפתח API אישי עבור Google Gemini ומפתח API אישי עבור Groq.
* ניהול הרשאות סנכרון ושמירה ל-MongoDB.

### 9.6 פאנל ניהול ועוקב חומרה (Admin Vitals Panel)
פאנל ייעודי עבור מנהל המערכת (Admin) המציג נתונים טכנולוגיים חיים:
* עוקב משאבים המציג את אחוז ניצול ה-CPU וניצול ה-RAM של שרת ה-Express המארח את המערכת בזמן אמת.
* רשימת Operatives (משתמשים רשומים במערכת) הכוללת מזהה ייחודי, שם וסטטוס פעילות.
