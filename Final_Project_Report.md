## חלק ב': אתגרים ופתרונות בתהליך הפיתוח (Challenges & Solutions)

במהלך הפיתוח נתקלתי במספר בעיות הנדסיות מורכבות, בעיקר בתחומי אבטחת מידע, ארכיטקטורת רשת, ואינטגרציה עם בינה מלאכותית. להלן הפירוט:

### 1. אתגר האבטחה: שמירה על פרטיות מפתח ה-API (Gemini Key)
* **הבעיה:** כדי שהמערכת תעבוד, השרת צריך לגשת למפתח ה-API של גוגל. שמירת המפתחות של המשתמשים בתוך מסד הנתונים היא סיכון אבטחה חמור – פריצה למסד תוביל לגניבת המפתחות ולחיובים כספיים עצומים למשתמשים.
* **הפתרון (BYOK Architecture):** פיתחתי ארכיטקטורת **"Bring Your Own Key"**. במקום לשמור את המפתח בשרת, המפתח נשמר בצורה מאובטחת ומקומית בלבד בתוך ה-`chrome.storage.local` של תוסף הכרום ושל הדפדפן. כאשר נשלחת בקשה לשרת, המפתח מצורף בתוך ה-HTTP Headers. השרת מקבל את הבקשה, משתמש במפתח באופן זמני בזיכרון ה-RAM כדי לבצע את הקריאה לגוגל, וזורק אותו מיד לאחר מכן. כך המפתח **מעולם לא נשמר** בשרת שלנו.

### 2. אתגר הסנכרון: תקשורת בין האתר, התוסף והשרת (JWT Auth)
* **הבעיה (תקלת התקשורת):** במהלך בדיקות התוסף, התקבלו שגיאות מהשרת: `Token is not valid` או `Unknown server error`. התברר שהשרת (Node.js) מוגן ב-Middleware הדורש JWT (JSON Web Token) תקף. מכיוון שהתוסף רץ מחוץ לאתר, לא היה לו את ה-Token של המשתמש המחובר, והשרת חסם את הבקשות ל-AI.
* **הפתרון:** פיתחתי מנגנון סנכרון רקע חכם בתוך ה-Extension. יצרתי קובץ `contentScript.js` שמוזרק לאתר הבית של J.A.R.V.I.S. הסקריפט בודק את ה-`localStorage` של הדפדפן כל שתי שניות, ואם הוא מזהה שהמשתמש מחובר (יש Token), הוא מעביר אותו בהודעה חשאית (`chrome.runtime.sendMessage`) אל ה-Background Worker של התוסף. כעת התוסף "יודע" מי המשתמש ומצרף את ה-Token העדכני לכל קריאה.

### 3. אתגר מיגרציית הנתונים: מעבר מ-Firebase ל-MongoDB
* **הבעיה:** במהלך הפיתוח הוחלט לעבור ממסד נתונים פשוט למסד נתונים מורכב וסקיילבילי (MongoDB Atlas). לאחר המעבר, דף ה-Dashboard שבו מוצגים הגרפים קרס לחלוטין עם השגיאה: `TypeError: act.createdAt.toDate is not a function`.
* **הפתרון:** מחקר קצר גילה שהשגיאה נובעת מהבדלי טיפוסי נתונים (Data Types). ב-Firebase, תאריכים נשמרו כאובייקט `Timestamp` שהכיל פונקציה מובנית `.toDate()`. לעומת זאת, MongoDB שומרת תאריכים כמחרוזת תקנית (ISO String). ביצעתי Refactoring לקוד צד הלקוח, הסרתי את כל קריאות ה-`.toDate()` והמרתי את הנתונים ישירות לאובייקט הזמן הסטנדרטי של JS בעזרת `new Date()`. האפליקציה חזרה לעבוד והגרפים הציגו נתונים מדויקים.

### 4. אתגר ה-AI: ניתוח ויזואלי של ממשקי משתמש (Computer Vision)
* **הבעיה:** פיתוח מסך ה-Screen Analysis דרש מהמערכת "לראות" תמונה ולהבין את העיצוב שלה. לא ניתן לשלוח קובץ תמונה רגיל בתוך גוף JSON ל-API של מודל שפה (LLM) פשוט.
* **הפתרון:** הטמעתי את המודל `gemini-1.5-flash` התומך ב-Multimodal Inputs. בצד הלקוח, המשתמש גורר תמונה, והמערכת סורקת וממירה אותה למחרוזת טקסטואלית ארוכה בפורמט **Base64**. המחרוזת נשלחת לשרת ב-POST Request. בשרת, הקוד מפענח את המחרוזת, מזהה את ה-MIME Type, ומרכיב אובייקט מסוג `inlineData`. האובייקט הזה מועבר למודל של גוגל יחד עם פרומפט מדויק שמאלץ את גוגל להחזיר רק JSON חוקי עם דירוג העיצוב, ללא טקסט חופשי. לבסוף, השרת מפרק את ה-JSON ושולח בחזרה למסך הלקוח נתונים מדויקים על בעיות ניגודיות, נגישות, ודירוג כללי.
